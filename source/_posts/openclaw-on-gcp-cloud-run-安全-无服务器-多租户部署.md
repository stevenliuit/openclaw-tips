<h2>前言</h2><p>几天前，Matias Kreder 发布了一篇关于在 AWS Bedrock AgentCore 上运行 OpenClaw 的精彩文章。这个架构非常优雅：临时容器、S3 支持的工作区同步、每用户隔离、无需持续运行的虚拟机。</p><p>我之前一直在 GKE 节点上运行 OpenClaw，账单还不错，但节点全天 24/7 运行，无论是否有人与智能体聊天。</p><p>阅读了 Matias 的文章后，我想：GCP 也有所有相同的原语。能否在 GCP 上原生复制这种模式？</p><p>答案是肯定的——而且在某些方面 GCP 路径更加简洁。Cloud Run v2 原生支持 GCSFuse 卷挂载，这意味着你可以在没有同步守护进程、sidecar 或后台定时器的情况下获得持久化的工作区。文件系统在容器重启后仍然正常工作。</p><p>这篇文章将介绍我如何在 Cloud Run 上构建多租户 OpenClaw 部署，具有完整的每租户隔离、Telegram/Slack 支持，以及共享路由服务作为唯一公共端点。完整代码仓库在 GitHub 上：openclaw-serverless。</p><h2>架构</h2><h3>GCSFuse 工作区持久化</h3><p>Cloud Run 容器是临时的——它们按需启动，空闲时消失。OpenClaw 将用户的所有信息存储在 .openclaw/ 下（对话记忆、用户资料、工具输出）。如果没有持久化策略，这些数据在会话结束的瞬间就会全部丢失。</p><p>这里的解决方案比 AWS 方案更简单：Cloud Run v2 内置了 GCSFuse 支持。智能体容器获得一个由每个租户 GCS 存储桶支持的 /data 卷挂载。入口点在启动时将 openclaw.json 写入该路径，智能体执行的每个文件写入都会透明地持久化到 GCS。没有同步循环，没有 SIGTERM 处理程序——它就是这样工作。容器重启后会从上次会话停止的地方继续。</p><p>一个刻意的细节：配置总是在启动时从环境变量覆盖。GCSFuse 持久化智能体状态；环境变量驱动配置。新的部署总是会覆盖磁盘上的过期配置。</p><h3>多租户路由</h3><p>我们没有将每个租户的 Cloud Run 服务暴露到互联网，而是让一个轻量级的 Node.js 路由位于公共端点。它验证 webhook 签名（Telegram 秘密令牌、Slack HMAC-SHA256），按用户/频道 ID 查找租户，然后使用 GCP 颁发的 ID 令牌将请求转发到正确的租户服务。租户服务使用 INGRESS_TRAFFIC_INTERNAL_ONLY 部署——它们完全无法通过路由访问。</p><p>Webhook 秘密从 Secret Manager 获取并缓存 5 分钟。两个通道都是故障关闭的：没有有效签名的请求在任何租户代码运行之前就会被拒绝。</p><h2>安全</h2><h3>网络</h3><p>租户 Cloud Run 服务只有内部入口。唯一的公共端点是路由服务。即使在 GCP 内，调用者也需要有效的 ID 令牌来调用租户服务——仅仅是网络访问是不够的。</p><h3>每租户隔离</h3><p>每个租户获得自己的 Cloud Run 服务、GCS 存储桶和服务账户。租户 SA 只有其自身存储桶的 objectAdmin——没有到任何其他租户资源的 IAM 绑定。秘密按租户范围划分；SA 可以访问自己的秘密加上共享的 Anthropic API 密钥，除此之外什么都不能访问。</p><h3>最小权限 IAM</h3><p>路由 SA 对 webhook 秘密有 secretAccessor，对每个租户服务有 run.invoker。租户 SA 对自己的秘密有 secretAccessor，对自己的存储桶有 objectAdmin。就这些。</p><h3>秘密管理</h3><p>Bot 令牌、webhook 秘密和 Anthropic API 密钥都存储在 Secret Manager 中。环境变量或容器镜像中没有敏感信息。</p><h3>设备配对绕过</h3><p>OpenClaw 通常需要交互式 shell 命令来批准设备。Cloud Run 没有 shell。dmPolicy: allowlist 与租户的用户 ID 在 allowFrom 中完全绕过配对——这是安全的，因为路由已经在消息到达之前验证了 webhook 来源。</p><h2>部署说明</h2><h3>前提条件</h3><ul><li>启用计费的 GCP 项目</li><li>gcloud CLI 已认证</li><li>已安装 terraform / opentofu</li><li>Docker 支持 linux/amd64 构建</li></ul><h3>1. 克隆仓库</h3><pre><code>git clone https://github.com/cloudon-one/openclaw-serverless cd openclaw-serverless</code></pre><h3>2. 配置项目</h3><pre><code>export PROJECT_ID=your-gcp-project-id export REGION=us-central1 export REGISTRY="$${REGION}-docker.pkg.dev/$${PROJECT_ID}/openclaw" gcloud config set project $$PROJECT_ID</code></pre><h3>3. 创建 Artifact Registry 仓库并启用 API</h3><pre><code>gcloud services enable run.googleapis.com secretmanager.googleapis.com artifactregistry.googleapis.com gcloud artifacts repositories create openclaw --repository-format=docker --location=$$REGION</code></pre><h3>4. 构建并推送两个容器镜像</h3><pre><code>./scripts/build.sh</code></pre><p>或手动：</p><pre><code>gcloud auth configure-docker $${REGION}-docker.pkg.dev docker build --platform linux/amd64 -t $${REGISTRY}/agent:latest agent/ docker build --platform linux/amd64 -t $${REGISTRY}/router:latest router/ docker push $${REGISTRY}/agent:latest docker push $${REGISTRY}/router:latest</code></pre><h3>5. 存储你的 Anthropic API 密钥</h3><pre><code>echo -n "YOUR_ANTHROPIC_API_KEY" | gcloud secrets create openclaw-anthropic-api-key --data-file=- --replication-policy=automatic</code></pre><h3>6. 在 tenants.yaml 中定义你的第一个租户</h3><pre><code>tenants:   alice:     display_name: "Alice Smith"     telegram_user_id: "YOUR_TELEGRAM_USER_ID"     telegram_enabled: true     slack_enabled: false     min_instances: 0     max_instances: 1     cpu: "2"     memory: "2Gi"</code></pre><h3>7. 部署基础设施</h3><pre><code>cd infrastructure cp terraform.tfvars.example terraform.tfvars terraform init terraform apply</code></pre><p>Terraform 创建：服务账户、GCS 存储桶、Secret Manager 容器、Cloud Run 服务（路由 + 每个租户一个）。</p><h3>8. 创建 Telegram Bot</h3><ul><li>在 Telegram 上向 @botfather 发送消息</li><li>使用 /newbot 并复制令牌</li></ul><h3>9. 存储租户秘密</h3><pre><code>echo -n "YOUR_BOT_TOKEN" | gcloud secrets versions add openclaw-sl-alice-telegram-token --data-file=- openssl rand -hex 32 | gcloud secrets versions add openclaw-sl-alice-telegram-webhook-secret --data-file=-</code></pre><h3>10. 注册 Telegram Webhook</h3><pre><code>ROUTER_URL=$$(cd infrastructure && terraform output -raw router_url) WEBHOOK_SECRET=$$(gcloud secrets versions access latest --secret=openclaw-sl-alice-telegram-webhook-secret) curl "https://api.telegram.org/bot$$YOUR_BOT_TOKEN/setWebhook" -d "url=$$ROUTER_URL/webhook/telegram" -d "secret_token=$$WEBHOOK_SECRET}"</code></pre><p>就是这样。在 Telegram 上向你的机器人发送消息。第一次响应需要约 15-20 秒进行冷启动；同一会话的后续消息会很快。</p><h2>总结</h2><p>这个方案效果很好，GCSFuse 方法确实比 S3 同步更简洁——更少的移动部件，没有 5 分钟的刷新窗口，没有关闭竞态条件。</p><p>在部署之前有几件事值得注意：</p><ul><li>cpu_idle: false 会增加成本，但是必需的。智能体会话涉及异步操作和 WebSocket 连接，在 CPU 节流下会中断。使用 min_instances: 0，你只在容器实际运行时付费，所以这是可以接受的——但不是免费的。</li><li>第二代执行环境是必需的。GCSFuse 在第一代中不可用。在 Terraform 中设置 execution_environment = "EXECUTION_ENVIRONMENT_GEN2"，否则挂载会静默失败。</li><li>冷启动是真实的。空闲租户的第一条消息需要 15-20 秒。对于异步聊天这没问题；对于任何延迟敏感的操作，这是个问题。如果需要，可以为每个租户设置 min_instances: 1——但要相应地预算。</li><li>添加第二个租户真的只是一个 YAML 条目和一次 terraform apply。</li></ul><p>隔离模型可以很好地扩展。每个租户都是一个完全独立的岛屿，没有共享状态。</p><p>Terraform 状态存储桶需要在 terraform init 之前存在。手动创建或单独引导——经典的先有鸡还是先有蛋的问题。</p><p>与 AWS AgentCore 方法相比，GCP 版本完全跳过了 NAT 网关（Cloud Run 有直接互联网出口），这消除了每月约 32 美元的 AWS 基线成本。对于单个个人智能体，这种架构在空闲时基本上是免费的。</p><p>想试试吗？代码仓库在 https://github.com/cloudon-one/openclaw-serverless 。如果你遇到问题或想扩展到其他通道（WhatsApp、Discord），路由很容易配置。</p>
