<h2>前言</h2><p>OpenClaw 以巨大的声势推出，我很好奇你是否真的可以"凭感觉编码"整个项目，特别是因为原始创建者使用 Codex 构建了它。我们正处于"自己构建而不是设置它"的时代，我想通过从零重建它来进一步推进这一理念。</p><p>这是如何使用现代编码智能体 SDK 重建 OpenClaw、解决多消息平台集成挑战并安全部署到生产环境的故事，同时避免了原始版本的安全陷阱。</p><h2>研究与规划</h2><p>我首先使用 GPT Pro 模式研究整个代码库并解释所有功能和使用工具。Pro 模型擅长处理大量信息的一次性综合任务。它给了我一份详细的产品规范，说明 OpenClaw 的工作原理以及每个功能使用的技术。</p><p>我决定使用编码智能体 SDK，因为它们代表了人们在写作之外首次真正使用 LLM 的真实用例。Claude 提供 Claude Agent SDK，OpenCode 提供类似的 SDK。这些 SDK 原生提供对 read、write、bash、edit 等工具的访问，并支持 skills 和 MCP（模型上下文协议）。</p><h2>架构概述</h2><p>我想要设置两种模式：</p><ul><li><strong>终端模式</strong>：用于直接交互和开发</li><li><strong>网关模式</strong>：用于 24/7 运行，监听 WhatsApp、Telegram、Signal、iMessage 和其他消息应用</li></ul><p>网关架构是 OpenClaw 强大的原因——它在后台持续运行，监控多个通信通道并自主响应。</p><h2>消息平台集成</h2><h3>WhatsApp</h3><p>WhatsApp 集成使用名为 Baileys 的库来建立 WhatsApp Web 连接。流程如下：</p><ul><li>Baileys 连接到 WhatsApp Web 的 WebSocket</li><li>当消息到达时，WhatsApp 服务器通过 WebSocket 推送它</li><li>Baileys 发出类型为 'notify' 的 messages.upsert 事件</li><li>智能体可以处理和响应消息</li></ul><p>我遇到的一个挑战是为 WhatsApp 号码创建白名单。WhatsApp 不直接在 WebSocket 连接中使用电话号码——它使用链接 ID。消息带有这些 ID，我需要在电话号码和链接 ID 之间进行双向转换。Claude Code 最初在构建正确的映射时遇到困难，但经过一些迭代后，我们正确地实现了它。</p><h3>Telegram</h3><p>由于其机器人 API，Telegram 更加简单。实现使用<strong>长轮询</strong>：</p><ul><li>定期调用 Telegram 的 getUpdates API</li><li>等待最多 30 秒获取新消息</li><li>当消息到达时，立即返回并再次调用 getUpdates</li><li>为每个新消息发出 message 事件</li></ul><p>机器人 API 文档完善，设置比 WhatsApp 容易得多。</p><h3>iMessage</h3><p>iMessage 集成是一个有趣的解锁。它使用名为 imsg 的库，由 Peter Steinberger 本人构建。方法是：</p><ul><li>读取存储所有 iMessage 的 SQLite 数据库</li><li>使用 FSEvents 监控数据库，这是 macOS 上内核级文件系统监控 API</li><li>当新消息写入数据库时实时检测</li></ul><p>这使得智能体无需任何官方 API 即可访问 iMessage。</p><h2>工具与集成</h2><p>正如所说，没有工具的智能体就什么都不是。我为智能体配备了：</p><p><strong>核心工具：</strong></p><ul><li>Read、Write、Edit（文件操作）</li><li>Bash（命令执行）</li><li>Glob、Grep（文件搜索）</li><li>TodoWrite（任务管理）</li><li>Skill（访问预定义工作流）</li><li>AskUserQuestion（用户交互）</li></ul><p><strong>自定义工具：</strong></p><ul><li>用于计划任务的 Cron 工具</li><li>用于 WhatsApp 和 Telegram 通信的网关工具</li></ul><p><strong>第三方集成：</strong></p><p>为了与 Slack、GitHub、Teams 等服务安全集成，我使用了 Composio。Composio 让你可以在沙箱环境中安全地连接和使用这些工具，同时处理所有凭证和身份验证。</p><h2>部署挑战</h2><h3>Docker 设置</h3><p>我创建了一个 Docker 设置，设计为在 DigitalOcean Droplet 后台运行。目标是使其可以快速部署，无需太多设置麻烦。然而，我遇到了几个问题：</p><p><strong>问题 1：OOM（内存溢出）错误</strong></p><p>在每月 6 美元的 2GB RAM 实例上运行，容器不断崩溃。问题是？它试图同时安装 Claude Code 和 OpenAI 的 SDK，耗尽了可用内存。确定后，我错开了安装，问题就解决了。</p><p><strong>问题 2：权限模式冲突</strong></p><p>网关使用 permissionMode: 'bypassPermissions' 以便智能体可以自主运行，无需为每个工具调用获得人工批准。然而，Claude Code 在以 root 运行时拒绝启用此功能——这是内置的安全功能。</p><p><strong>解决方案：</strong></p><p>我不得不重构整个 Dockerfile 以使用非 root 用户：</p><pre><code># 创建非 root 用户（Claude Code 拒绝以 root 身份使用 bypassPermissions）
RUN useradd -m -s /bin/bash claw && chown -R claw:claw /app
USER claw</code></pre><p>这导致了级联修复：</p><ul><li>所有文件路径（/root/ → /home/claw/）</li><li>Docker Compose 卷挂载</li><li>CLI 安装目录</li><li>工作区权限</li></ul><p>重构花了几个小时，但结果是一个更安全的部署，遵循最佳实践。</p><h2>关键要点</h2><ul><li><strong>现代编码智能体非常强大</strong>——凭借适当的工具和上下文，它们可以从零重建复杂系统</li><li><strong>设计时考虑安全很重要</strong>——强制的非 root 用户设置，虽然最初令人沮丧，但导致了更安全的架构</li><li><strong>集成复杂性差异很大</strong>——Telegram 花了 30 分钟，WhatsApp 花了几个小时，iMessage 需要创造性解决方案</li><li><strong>资源限制迫使更好的架构</strong>——2GB RAM 限制促使我优化安装和运行时行为</li><li><strong>文档就是一切</strong>——拥有良好 API 的服务（如 Telegram）比需要逆向工程的服务更容易集成</li></ul><h2>下一步</h2><p>重建的 OpenClaw 现在已在生产中运行，处理跨多个平台的消息，没有困扰原始版本的安全问题。未来的改进包括：</p><ul><li>添加更多消息平台（Discord、Slack DM）</li><li>实施更好的错误处理和重试逻辑</li><li>创建用于监控和配置的 Web 仪表板</li><li>优化内存使用以在更小的实例上运行</li></ul><p>从零构建这是理解现代 AI 智能体如何在生产中工作的绝佳练习。LLM 能力、适当工具和精心架构的结合使创建以前难以构建的强大自主系统成为可能。</p>
