<h2>前言</h2><p><strong>本系列第一部分：</strong>在AWS Lightsail上部署自主AI智能体的安全设置、真实发现和攻击面分析。</p><p>AWS刚刚宣布了<strong>OpenClaw on Amazon Lightsail</strong>的全面可用性——一个开源、自托管的自主AI智能体，连接到WhatsApp、Telegram、Discord，并独立执行任务：运行代码、管理文件、浏览网页。</p><p>社区都在测试它。我也不例外——但是以<strong>云安全工程师</strong>的视角。</p><p>这篇文章不是关于如何使用OpenClaw作为助手，而是关于<strong>我从安全角度发现的设置问题、我的决策以及为什么这样做</strong>。</p><h2>OpenClaw到底是什么？</h2><p>在讨论安全之前，让我们先对齐概念。</p><p><strong>LLM</strong>（如Claude或GPT）接收提示并返回文本。就这么回事。</p><p><strong>自主智能体</strong>则不同：它可以访问工具（终端、浏览器、API），决定使用它们的顺序，解释结果并采取行动——无需你 directing each step。LLM是"大脑"，但智能体是能够在现实世界中运作的完整系统。</p><p><strong>OpenClaw正是这样：</strong>一个在你的服务器上运行的智能体，使用LLM（通过<strong>Amazon Bedrock</strong>）作为推理引擎，能够通过消息通道自主执行任务。</p><p>这种自主性正是它的强大之处，也是从安全角度有趣的地方。</p><h2>Lightsail设置</h2><p>AWS将OpenClaw打包成Lightsail蓝图——这意味着你可以在几分钟内启动一个运行实例，无需手动配置。</p><h3>我在向导中看到的（以及引起我注意的）</h3><p><strong>SSH密钥对：</strong></p><p>Lightsail给你两个选项：让AWS生成密钥对，或上传你自己的。</p><p>安全建议很明确：<strong>在本地生成</strong>自己的。</p><pre><code>ssh-keygen -t ed25519 -C "openclaw-sandbox"</code></pre><p>为什么？当AWS生成密钥对时，私钥是在他们的服务器上创建的，然后传输给你下载。那一刻的传输是一个可以避免的风险。如果你自己生成，私钥<strong>永远不会离开你的机器</strong>。</p><p>类比：公钥是你放在服务器上的锁。私钥是你唯一持有的钥匙。任何人都能看到锁，但没有人能打开它。</p><blockquote><p>💡<strong>记住：</strong>始终在私钥上运行<code>chmod 400</code>。如果是<code>644</code>权限，其他系统用户可以读取它——SSH客户端本身会警告你。</p></blockquote><p><strong>防火墙（安全组）：</strong></p><p>默认情况下，Lightsail向<code>0.0.0.0/0</code>（世界上的任何IP）开放端口80、443和22。</p><p>对于个人沙箱来说没必要。我将所有三条规则限制为<strong>仅对我的IP开放</strong>：</p><pre><code>curl ifconfig.me # get your public IP</code></pre><p>减少暴露的攻击面 = 如果出问题时的更小爆炸半径。</p><h2>第一个发现：过时的操作系统</h2><p>当我通过SSH连接时，欢迎信息很直接：</p><pre><code>44 updates can be applied immediately. 31 of these updates are standard security updates.</code></pre><p>AWS蓝图附带<strong>31个未打的安全更新</strong>。包括内核补丁和<code>intel-microcode</code>更新。</p><p>为什么内核很重要？因为它是操作系统的核心——控制内存、进程和权限。已知的内核漏洞如<strong>Dirty Pipe</strong>或<strong>Spectre/Meltdown</strong>允许权限提升或从其他进程读取内存。</p><p>在运行Docker容器（如OpenClaw）的服务器上，未打补丁的内核可能被利用进行<strong>容器逃逸</strong>——突破隔离容器并完全控制主机。</p><p>修复很简单：</p><pre><code>sudo apt update && sudo apt upgrade -y sudo reboot</code></pre><blockquote><p>🔍<strong>发现 #1：</strong>OpenClaw蓝图在Lightsail上部署时内核和系统库过时。任何在生产环境中使用它而不打补丁的客户都会暴露在已知CVE风险中。</p></blockquote><h2>最有趣的部分：OpenClaw安全设置</h2><p>设置后，OpenClaw呈现5个安全配置。这是有趣的部分。</p><pre><code>1. File & folder protection → Status: ✓ Protected
2. Browser remote control → Status: ✓ Disabled
3. Exec host policy → Status: ~ Not set (defaults to sandbox)
4. Shell command approval → Status: ~ Unrestricted
5. Access token → Status: ~ Never rotated</code></pre><h3>设置3：Exec主机策略</h3><p>这控制智能体<strong>在哪里</strong>执行shell命令。</p><ul><li><code>sandbox</code>（默认）：命令在隔离的Docker容器内运行。</li><li><code>gateway</code>：命令直接在服务器操作系统上运行。</li></ul><p>爆炸半径差异巨大。使用<code>gateway</code>，如果智能体被 compromise，攻击者可以直接访问文件系统、环境变量、IAM角色——一切。</p><h3>设置4：Shell命令审批</h3><p>这控制智能体在执行命令<strong>是否需要请求许可</strong>。</p><ul><li><code>deny</code>：阻止所有shell命令。</li><li><code>allow</code>：自由执行，不询问。</li></ul><h3>你不能忽视的攻击链</h3><p>这是最危险的场景，如果你没有正确配置的话，完全可能发生：</p><pre><code>exec host policy: gateway + shell command approval: allow</code></pre><p><strong>结果：</strong>智能体可以直接在服务器上执行任何命令，没有隔离，也没有请求许可。</p><p>在此基础上添加<strong>提示词注入</strong>攻击——攻击者将恶意指令插入智能体消费的内容中（邮件、文件、网页）——攻击者可以在你不知情的情况下完全控制服务器。</p><p>这种弱配置链接称为<strong>攻击链</strong>，这正是智能体投入生产前需要进行的分析类型。</p><blockquote><p>🔍<strong>发现 #2：</strong><code>exec host policy: gateway</code> + <code>shell command approval: allow</code>的组合消除了所有智能体隔离层。在生产环境中，这代表严重风险。</p></blockquote><h3>设置5：仪表板中可见的Gateway Token</h3><p>Gateway Token是智能体的"密码"——谁拥有它就完全控制OpenClaw。</p><p><strong>问题：</strong>它在仪表板中以明文显示。</p><p>任何仪表板截图、任何屏幕录制、任何人在你身后看——都会 compromise 智能体的访问。</p><p>AWS建议经常轮换它，不要在配置文件中硬编码。但它在UI中默认可见这一事实是一个值得注意的设计考虑。</p><blockquote><p>🔍<strong>发现 #3：</strong>Gateway Token在仪表板中以明文显示。结合暴露在互联网上的仪表板，这代表直接的凭证暴露。</p></blockquote><h2>默认启用IPv6</h2><p>一个容易被忽视的细节：蓝图默认启用了<strong>双栈（IPv4 + IPv6）</strong>。</p><p>安全问题：许多防火墙规则是针对IPv4编写的。如果你不检查两种协议族，IPv6流量可能不会被注意到。</p><p>如果你不使用IPv6，请禁用它。不必要的攻击面。</p><h2>这个蓝图安装了什么？</h2><p>在<code>apt upgrade</code>后重启的服务中，一个有趣的服务出现了：</p><pre><code>systemctl restart apache2.service</code></pre><p>OpenClaw仪表板运行在<strong>Apache2</strong>上。这意味着Apache是另一个需要考虑的攻击面——有它自己的CVE和配置需要审查。</p><p>熟悉Apache漏洞的攻击者可能尝试绕过网关身份验证，而根本不需要token。</p><h2>发现总结</h2><table><tr><th>#</th><th>发现</th><th>严重程度</th></tr><tr><td>1</td><td>蓝图部署时内核和系统库过时（31个待处理安全更新）</td><td>高</td></tr><tr><td>2</td><td>gateway + allow组合消除隔离和命令审批</td><td>严重</td></tr><tr><td>3</td><td>Gateway Token在仪表板中以明文显示</td><td>高</td></tr><tr><td>4</td><td>IPv6默认启用，向导中无禁用选项</td><td>中</td></tr><tr><td>5</td><td>Apache2作为Web服务器，无文档化的加固</td><td>中</td></tr></table><h2>安全配置建议</h2><p>如果你在真实环境中部署OpenClaw：</p><ul><li><strong>本地生成密钥对</strong>——永远不要让提供商帮你生成。</li><li><strong>限制防火墙</strong>——从一开始就将访问限制到已知IP。</li><li><strong>立即打补丁</strong>——部署后不要信任蓝图是最新的。</li><li>保持<code>exec host policy</code>在<code>sandbox</code>——Docker隔离是你的第一道防线。</li><li>将<code>shell command approval</code>设置为<code>deny</code>或需要明确审批——对于不可逆的操作需要人工介入。</li><li><strong>定期轮换Gateway Token</strong>，永远不要在截图中暴露它。</li><li><strong>禁用IPv6</strong>——如果你不使用它。</li><li><strong>不要将仪表板暴露在互联网上</strong>——如果你需要远程访问，使用VPN或安全隧道。</li></ul><h2>为什么这很重要</h2><p>自主AI智能体正在LATAM各地的公司投入生产。OpenClaw只是第一个。</p><p>大多数部署它们的团队没有考虑攻击面、配置错误导致的爆炸半径，或者提示词注入如何与配置错误链接以完全 compromise 服务器。</p><p>这正是云安全工程师需要介入的地方。</p><h2>下一步</h2><p>在<strong>第二部分</strong>中，我将探索我们今天没有触及的领域：渠道（WhatsApp/Telegram集成）、智能体、Cron任务，以及每个如何扩展系统的攻击面。</p><p>我还将使用<strong>OWASP Top 10 for Agentic
